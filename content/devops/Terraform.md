## 1. What is Terraform

Terraform is an open-source **Infrastructure as Code (IaC)** tool created by HashiCorp that lets you define, provision, and manage cloud and on-premises infrastructure using declarative configuration files.

**Declarative** means you describe the *desired end state* of your infrastructure, and Terraform figures out the steps to get there. You don't write "create this, then create that" — you write "this should exist," and Terraform reconciles reality with your specification.

### Key Characteristics

- **Cloud-agnostic** — One tool for AWS, Azure, GCP, Oracle Cloud, Kubernetes, and 3,000+ other providers
- **Stateful** — Tracks what it has provisioned in a state file
- **Plan-before-apply** — Always shows you what will change before making changes
- **Idempotent** — Running the same config twice produces the same result
- **Resource graph** — Builds a dependency graph to parallelize operations safely

### Note on Licensing

In August 2023, HashiCorp moved Terraform from the Mozilla Public License (MPL) to the Business Source License (BSL). This led the community to fork the last MPL version into **OpenTofu**, now maintained by the Linux Foundation. Both tools are largely compatible. For most users and most use cases, the choice doesn't matter day-to-day.



## 2. Why Terraform

### The Problem It Solves

Before IaC, infrastructure was managed by:
- Clicking through cloud consoles (not reproducible)
- Writing shell scripts with cloud CLIs (imperative, hard to update)
- Using cloud-specific tools like CloudFormation (vendor lock-in)

### The Benefits

| Benefit | What It Means in Practice |
|||
| **Reproducibility** | Spin up identical dev/staging/prod environments |
| **Version control** | Every infra change is a Git commit you can audit and revert |
| **Code review** | Infrastructure changes get the same scrutiny as application code |
| **Documentation** | The code *is* the documentation |
| **Disaster recovery** | Rebuild your entire infrastructure from code in minutes |
| **Multi-cloud** | Manage AWS, Azure, GCP, Cloudflare, GitHub all from one tool |
| **Cost visibility** | Tools like Infracost show the dollar impact of every PR |



## 3. Core Concepts

### Resources
The fundamental building block — anything Terraform can create, read, update, or delete.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
}
```

### Providers
Plugins that let Terraform talk to a specific platform's API (AWS, Azure, Kubernetes, GitHub, Datadog, etc.).

### State
A JSON file (`terraform.tfstate`) that maps your config to real-world resources. **This is critical** — losing it means Terraform forgets what it provisioned.

### Plan
A preview of changes Terraform will make. Generated by comparing config + state against actual cloud reality.

### Apply
The execution of a plan — actually creating, updating, or destroying resources.

### Modules
Reusable, parameterized bundles of Terraform code. The closest analog is a function in programming.

### Backends
Where state is stored (local disk, S3, Azure Storage, Terraform Cloud, GCS, etc.).

### Workspaces
Isolated state environments within a single configuration (typically used for dev/staging/prod).

### Data Sources
Read-only references to existing infrastructure not managed by your Terraform config.

```hcl
data "aws_vpc" "default" {
  default = true
}
```


## 6. HCL Syntax Essentials

HashiCorp Configuration Language (HCL) is purpose-built for infrastructure declarations.

### Block Structure
```hcl
block_type "label_one" "label_two" {
  argument = value
  
  nested_block {
    nested_argument = value
  }
}
```

### Variables
```hcl
# variables.tf
variable "region" {
  description = "AWS region for resources"
  type        = string
  default     = "ap-south-1"
  
  validation {
    condition     = contains(["ap-south-1", "us-east-1", "eu-west-1"], var.region)
    error_message = "Region must be one of the approved values."
  }
}

variable "instance_count" {
  type    = number
  default = 2
}

variable "tags" {
  type = map(string)
  default = {
    Environment = "dev"
    ManagedBy   = "terraform"
  }
}

variable "allowed_cidrs" {
  type    = list(string)
  default = ["10.0.0.0/8"]
}

variable "database_config" {
  type = object({
    engine         = string
    instance_class = string
    storage_gb     = number
    multi_az       = bool
  })
  default = {
    engine         = "postgres"
    instance_class = "db.t3.medium"
    storage_gb     = 100
    multi_az       = false
  }
}

variable "db_password" {
  type      = string
  sensitive = true
}
```

### Outputs
```hcl
# outputs.tf
output "vpc_id" {
  description = "ID of the created VPC"
  value       = aws_vpc.main.id
}

output "database_endpoint" {
  value     = aws_db_instance.main.endpoint
  sensitive = true
}
```

### Locals
```hcl
locals {
  common_tags = {
    Project     = "platform"
    Environment = var.environment
    Owner       = "platform-team"
  }
  
  name_prefix = "${var.project}-${var.environment}"
}
```

### Expressions and Functions
```hcl
# String interpolation
name = "${var.environment}-app-server"

# Conditional
instance_type = var.environment == "prod" ? "m5.large" : "t3.micro"

# Functions
upper_name      = upper(var.name)
joined_list     = join(",", var.subnets)
file_contents   = file("${path.module}/script.sh")
parsed_json     = jsondecode(file("config.json"))
formatted_date  = formatdate("YYYY-MM-DD", timestamp())
cidr_subnets    = cidrsubnet("10.0.0.0/16", 8, 1)  # 10.0.1.0/24
```

### Loops with for_each
```hcl
resource "aws_iam_user" "team" {
  for_each = toset(["alice", "bob", "charlie"])
  name     = each.key
}

resource "aws_s3_bucket" "buckets" {
  for_each = {
    logs    = "company-logs-bucket"
    backups = "company-backups-bucket"
    assets  = "company-assets-bucket"
  }
  
  bucket = each.value
  tags = {
    Purpose = each.key
  }
}
```

### Loops with count
```hcl
resource "aws_instance" "workers" {
  count         = var.worker_count
  ami           = var.ami_id
  instance_type = "t3.medium"
  
  tags = {
    Name = "worker-${count.index}"
  }
}
```

> **Tip:** Prefer `for_each` over `count`. With `count`, removing an item shifts indices and causes resources to be destroyed and recreated.

### Dynamic Blocks
```hcl
resource "aws_security_group" "web" {
  name = "web-sg"
  
  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.from_port
      to_port     = ingress.value.to_port
      protocol    = ingress.value.protocol
      cidr_blocks = ingress.value.cidr_blocks
    }
  }
}
```



## 7. The Complete Command Reference

### Initialization
```bash
terraform init                          # Initialize working directory
terraform init -upgrade                 # Upgrade providers to latest matching versions
terraform init -reconfigure             # Reconfigure backend, ignoring saved config
terraform init -migrate-state           # Migrate state to new backend
terraform init -backend-config=file.hcl # Use partial backend config
```

### Planning
```bash
terraform plan                          # Preview changes
terraform plan -out=tfplan              # Save plan to file
terraform plan -var="region=us-east-1"  # Override variable
terraform plan -var-file="prod.tfvars"  # Use specific vars file
terraform plan -target=aws_instance.web # Plan only specific resource
terraform plan -destroy                 # Preview a destroy operation
terraform plan -refresh-only            # Detect drift without changes
```

### Applying
```bash
terraform apply                         # Apply changes (prompts for confirmation)
terraform apply tfplan                  # Apply a saved plan
terraform apply -auto-approve           # Skip confirmation (use carefully!)
terraform apply -parallelism=20         # Increase concurrent operations
terraform apply -replace=aws_instance.web  # Force replacement of a resource
```

### Destruction
```bash
terraform destroy                       # Tear down everything
terraform destroy -target=aws_instance.web  # Destroy specific resource
terraform plan -destroy -out=destroy.tfplan # Save destroy plan
```

### State Inspection
```bash
terraform show                          # Show current state
terraform show -json                    # Machine-readable output
terraform state list                    # List all resources in state
terraform state show aws_instance.web   # Show specific resource
terraform output                        # Show all outputs
terraform output -json database_url     # Specific output as JSON
terraform output -raw api_key           # Raw value (no quotes)
```

### State Manipulation
```bash
terraform state mv old_name new_name    # Rename a resource in state
terraform state rm aws_instance.web     # Remove from state (won't destroy resource)
terraform state pull > backup.tfstate   # Download remote state
terraform state push backup.tfstate     # Upload state (dangerous!)
terraform import aws_instance.web i-1234567890  # Bring existing resource into state
terraform refresh                       # Update state to match real infrastructure
```

### Validation and Formatting
```bash
terraform fmt                           # Format files in current directory
terraform fmt -recursive                # Format all .tf files recursively
terraform fmt -check                    # Check formatting (CI use)
terraform fmt -diff                     # Show formatting diffs
terraform validate                      # Check syntax and internal consistency
```

### Workspaces
```bash
terraform workspace list                # List all workspaces
terraform workspace new dev             # Create new workspace
terraform workspace select prod         # Switch workspace
terraform workspace show                # Show current workspace
terraform workspace delete old-env      # Delete workspace
```

### Modules
```bash
terraform get                           # Download/update modules
terraform get -update                   # Force update modules
```

### Debugging
```bash
TF_LOG=DEBUG terraform apply            # Enable debug logs
TF_LOG=TRACE terraform apply            # Maximum verbosity
TF_LOG_PATH=./tf.log terraform apply    # Send logs to file
terraform console                       # Interactive REPL for testing expressions
```

### Graph and Visualization
```bash
terraform graph | dot -Tpng > graph.png  # Visualize dependency graph (requires graphviz)
terraform graph -type=plan
```

### Workspace Cleanup
```bash
rm -rf .terraform/                       # Remove provider plugins and modules
rm .terraform.lock.hcl                   # Remove provider lockfile
```



## 8. State Management

State is the most operationally critical aspect of Terraform. Master this and you'll avoid 80% of production headaches.

### Why State Exists
Terraform needs to know:
- Which resources it manages
- The mapping between config and real cloud resources
- Resource metadata (IDs, ARNs, computed attributes)
- Dependencies between resources

### Local vs Remote State

**Local state** (`terraform.tfstate` on disk) — fine for solo learning, **never** for teams.

**Remote state** is mandatory in production because:
- Multiple team members can collaborate
- State is backed up automatically
- Locking prevents concurrent modifications
- Sensitive values aren't in your local filesystem

### Remote Backend: AWS S3 + DynamoDB
```hcl
# backend.tf
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "infrastructure/prod/terraform.tfstate"
    region         = "ap-south-1"
    encrypt        = true
    dynamodb_table = "terraform-state-locks"
  }
}
```

The DynamoDB table provides state locking. Create it with:
```hcl
resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  
  attribute {
    name = "LockID"
    type = "S"
  }
}
```

### Remote Backend: Azure Storage
```hcl
terraform {
  backend "azurerm" {
    resource_group_name  = "tfstate"
    storage_account_name = "tfstateaccount"
    container_name       = "tfstate"
    key                  = "prod.terraform.tfstate"
  }
}
```

### Remote Backend: Terraform Cloud
```hcl
terraform {
  cloud {
    organization = "my-org"
    workspaces {
      name = "production"
    }
  }
}
```

### State Locking
When someone runs `terraform apply`, Terraform acquires a lock. Others trying to apply see:
```
Error: Error acquiring the state lock
Lock Info:
  ID:        abc-123
  Path:      bucket/path/terraform.tfstate
  Operation: OperationTypeApply
  Who:       alice@laptop
```

Force-unlock only when you're certain the holder crashed:
```bash
terraform force-unlock abc-123
```

### State Drift
Drift happens when someone modifies infrastructure outside Terraform (manual console changes, other tools).

Detect drift:
```bash
terraform plan -refresh-only
```

Reconcile by either:
1. Updating Terraform code to match reality
2. Running `terraform apply` to revert manual changes

### Importing Existing Resources
You inherit a manually-created EC2 instance and want it under Terraform management:

```hcl
# 1. Write the resource block
resource "aws_instance" "legacy_server" {
  # Leave this minimal initially
  ami           = "ami-xxx"
  instance_type = "t3.medium"
}
```

```bash
# 2. Import existing resource into state
terraform import aws_instance.legacy_server i-0abc123def456

# 3. Run plan to see what's different between your config and reality
terraform plan

# 4. Update your code to match reality, until plan shows no changes
```

### Modern Import Blocks (Terraform 1.5+)
```hcl
import {
  to = aws_instance.legacy_server
  id = "i-0abc123def456"
}

resource "aws_instance" "legacy_server" {
  ami           = "ami-xxx"
  instance_type = "t3.medium"
}
```

Generate config automatically:
```bash
terraform plan -generate-config-out=generated.tf
```

### State Splitting (When State Files Get Too Big)
Symptoms: plan/apply takes 10+ minutes, blast radius too large.

Solution: split into multiple smaller state files by domain:
```
infrastructure/
├── networking/      # VPCs, subnets — rarely changes
├── data/            # Databases — careful changes
├── compute/         # EC2, EKS — frequent changes
└── application/     # App-specific resources
```

Use `terraform_remote_state` data source to read between them:
```hcl
data "terraform_remote_state" "networking" {
  backend = "s3"
  config = {
    bucket = "company-terraform-state"
    key    = "networking/terraform.tfstate"
    region = "ap-south-1"
  }
}

resource "aws_instance" "app" {
  subnet_id = data.terraform_remote_state.networking.outputs.private_subnet_id
}
```



## 9. Modules

Modules are how you scale Terraform from one-off scripts to a production codebase.

### Anatomy of a Module
```
modules/vpc/
├── main.tf          # Resources
├── variables.tf     # Input parameters
├── outputs.tf       # Exported values
├── versions.tf      # Version constraints
└── README.md        # Usage documentation
```

### Example Module: VPC

**modules/vpc/variables.tf**
```hcl
variable "name" {
  type        = string
  description = "Name prefix for VPC resources"
}

variable "cidr_block" {
  type        = string
  description = "CIDR block for VPC"
  default     = "10.0.0.0/16"
}

variable "availability_zones" {
  type        = list(string)
  description = "List of AZs to deploy subnets in"
}

variable "tags" {
  type    = map(string)
  default = {}
}
```

**modules/vpc/main.tf**
```hcl
resource "aws_vpc" "main" {
  cidr_block           = var.cidr_block
  enable_dns_hostnames = true
  enable_dns_support   = true
  
  tags = merge(var.tags, {
    Name = var.name
  })
}

resource "aws_subnet" "public" {
  count = length(var.availability_zones)
  
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.cidr_block, 8, count.index)
  availability_zone = var.availability_zones[count.index]
  
  tags = merge(var.tags, {
    Name = "${var.name}-public-${count.index}"
    Type = "public"
  })
}
```

**modules/vpc/outputs.tf**
```hcl
output "vpc_id" {
  value = aws_vpc.main.id
}

output "public_subnet_ids" {
  value = aws_subnet.public[*].id
}
```

### Using a Module
```hcl
module "production_vpc" {
  source = "./modules/vpc"
  
  name               = "prod"
  cidr_block         = "10.0.0.0/16"
  availability_zones = ["ap-south-1a", "ap-south-1b", "ap-south-1c"]
  
  tags = {
    Environment = "production"
  }
}

# Reference module outputs
resource "aws_instance" "web" {
  subnet_id = module.production_vpc.public_subnet_ids[0]
}
```

### Module Sources
```hcl
# Local path
source = "./modules/vpc"

# Terraform Registry
source  = "terraform-aws-modules/vpc/aws"
version = "~> 5.0"

# GitHub
source = "github.com/my-org/terraform-modules//vpc?ref=v1.2.0"

# Git over SSH
source = "git::ssh://git@github.com/my-org/terraform-modules.git//vpc?ref=v1.2.0"

# S3
source = "s3::https://s3.amazonaws.com/my-bucket/vpc.zip"

# HTTP archive
source = "https://example.com/vpc-module.tar.gz"
```

### Always Pin Module Versions
```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"  # Exact version for production
}
```

### Module Composition Pattern
Build small, focused modules and compose them:
```hcl
module "network" {
  source = "./modules/network"
  # ...
}

module "kubernetes" {
  source = "./modules/eks"
  
  vpc_id     = module.network.vpc_id
  subnet_ids = module.network.private_subnet_ids
}

module "database" {
  source = "./modules/rds"
  
  vpc_id     = module.network.vpc_id
  subnet_ids = module.network.database_subnet_ids
}
```



## 10. Workspaces and Environments

### Two Approaches

**Approach 1: Workspaces (simpler, but limited)**
```bash
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod
terraform workspace select prod
terraform apply
```

```hcl
resource "aws_instance" "web" {
  instance_type = terraform.workspace == "prod" ? "m5.large" : "t3.micro"
  
  tags = {
    Environment = terraform.workspace
  }
}
```

**Limitation:** All workspaces share the same backend config and same code. Risky if you need significantly different setups.

**Approach 2: Directory-per-environment (recommended for prod)**
```
environments/
├── dev/
│   ├── main.tf            # Calls modules with dev params
│   ├── backend.tf         # dev state location
│   └── terraform.tfvars
├── staging/
└── prod/
```

**environments/prod/main.tf**
```hcl
module "platform" {
  source = "../../modules/platform"
  
  environment    = "prod"
  instance_type  = "m5.xlarge"
  instance_count = 6
  enable_ha      = true
}
```

**environments/prod/backend.tf**
```hcl
terraform {
  backend "s3" {
    bucket = "company-terraform-state"
    key    = "prod/terraform.tfstate"
    region = "ap-south-1"
  }
}
```

This pattern is safer because:
- Different environments can have different module versions
- State files are completely isolated
- Permissions can differ per environment
- Lower blast radius for mistakes



## 11. Providers Deep Dive

### Provider Configuration
```hcl
terraform {
  required_version = ">= 1.5.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.20"
    }
  }
}

provider "aws" {
  region = "ap-south-1"
  
  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Project     = "platform"
    }
  }
}
```

### Multiple Provider Instances (Aliases)
```hcl
provider "aws" {
  region = "ap-south-1"
  alias  = "mumbai"
}

provider "aws" {
  region = "us-east-1"
  alias  = "virginia"
}

resource "aws_instance" "primary" {
  provider = aws.mumbai
  # ...
}

resource "aws_instance" "dr_replica" {
  provider = aws.virginia
  # ...
}
```

### Cross-Account AWS
```hcl
provider "aws" {
  region = "ap-south-1"
  
  assume_role {
    role_arn     = "arn:aws:iam::123456789012:role/TerraformExecution"
    session_name = "terraform-session"
  }
}
```

### GitHub Actions Workflow
```yaml
# .github/workflows/terraform.yml
name: Terraform

on:
  pull_request:
    paths: ['infrastructure/**']
  push:
    branches: [main]
    paths: ['infrastructure/**']

permissions:
  contents: read
  pull-requests: write
  id-token: write   # For AWS OIDC

jobs:
  terraform:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./infrastructure/environments/prod
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActionsTerraform
          aws-region: ap-south-1
      
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.6.0
      
      - name: Terraform Format Check
        run: terraform fmt -check -recursive
      
      - name: Terraform Init
        run: terraform init
      
      - name: Terraform Validate
        run: terraform validate
      
      - name: tfsec Security Scan
        uses: aquasecurity/tfsec-action@v1.0.0
      
      - name: Terraform Plan
        if: github.event_name == 'pull_request'
        id: plan
        run: terraform plan -no-color -out=tfplan
        continue-on-error: true
      
      - name: Comment Plan on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const output = `#### Terraform Plan
            \`\`\`
            ${{ steps.plan.outputs.stdout }}
            \`\`\``;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });
      
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main'
        run: terraform apply -auto-approve
```

## 13. Security and Secrets

### Never Commit Secrets

❌ **Wrong:**
```hcl
resource "aws_db_instance" "main" {
  password = "MyP@ssw0rd123"  # Hardcoded — never do this
}
```

✅ **Right:**
```hcl
resource "aws_db_instance" "main" {
  password = var.db_password
}

variable "db_password" {
  type      = string
  sensitive = true
}
```

### Pulling Secrets from Vault
```hcl
data "vault_generic_secret" "db_creds" {
  path = "secret/data/database/prod"
}

resource "aws_db_instance" "main" {
  username = data.vault_generic_secret.db_creds.data["username"]
  password = data.vault_generic_secret.db_creds.data["password"]
}
```

### AWS Secrets Manager
```hcl
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/db/password"
}

resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
```

### Sensitive Outputs
```hcl
output "database_password" {
  value     = random_password.db.result
  sensitive = true
}
```

Sensitive values are masked in CLI output but **still appear in state files**. Encrypt your state backend.

### IAM Best Practices for Terraform Execution
- Use **OIDC federation** for CI/CD instead of long-lived access keys
- Grant least-privilege IAM policies
- Separate IAM roles per environment
- Enable CloudTrail to audit all Terraform operations

## 15. Real-World Scenarios

### Scenario 1: Onboarding to an Existing Terraform Codebase

**Day 1 checklist:**
1. Clone repo, read top-level README
2. Identify the backend (where's state stored?)
3. Get IAM credentials with read access first
4. Run `terraform init` then `terraform plan` — there should be **zero changes**
5. If there ARE changes, someone has drifted infra or forgotten to merge code
6. Map the module structure before changing anything
7. Make a tiny, low-risk change (add a tag) as your first PR

### Scenario 2: "We Have a Production Outage and State Is Locked"

```bash
# 1. Identify the lock holder
terraform plan
# Shows: Lock ID: abc-123, Who: ci-runner-42

# 2. Check if that process is still running (maybe CI runner crashed)
# Check your CI system, check active sessions

# 3. If confirmed dead, force-unlock
terraform force-unlock abc-123

# 4. Add this incident to your runbook
```

### Scenario 3: Someone Manually Changed a Resource

```bash
# Detect drift
terraform plan -refresh-only

# Two options:
# A) Code wins — overwrite manual change
terraform apply

# B) Manual change is correct — update code to match
# Edit .tf files, then plan should show no changes
```

### Scenario 4: Renaming a Resource Without Recreating It

You want to rename `aws_instance.old_name` to `aws_instance.web_server` without destroying the EC2 instance.

```bash
terraform state mv aws_instance.old_name aws_instance.web_server
```

Then update your code. `terraform plan` should show no changes.

### Scenario 5: Splitting a Monolith State

```bash
# 1. Backup current state
terraform state pull > backup.tfstate

# 2. In source workspace, remove resources from state
terraform state rm module.networking

# 3. In new workspace, import them
cd ../networking-workspace/
terraform import module.networking.aws_vpc.main vpc-1234567

# 4. Verify both workspaces show no changes
```

### Scenario 6: Provisioning a New Environment

```bash
# 1. Copy template environment
cp -r environments/staging environments/uat

# 2. Update backend.tf with new state path
# 3. Update terraform.tfvars with environment-specific values
# 4. Update any env-specific module inputs

cd environments/uat
terraform init
terraform plan       # Review carefully
terraform apply
```

### Scenario 7: Disaster Recovery Drill

```bash
# Goal: rebuild production in another region in <30 minutes

# 1. Backup state file (versioned in S3 already)
aws s3 cp s3://company-terraform-state/prod/terraform.tfstate ./

# 2. Have DR environment ready in another region with separate state
cd environments/dr-mumbai

# 3. Run apply (resources are pre-defined, just need to provision)
terraform apply -auto-approve

# 4. Run validation tests
./scripts/validate-dr.sh

# 5. Update DNS to point to DR
# (this can also be in Terraform!)
```

### Scenario 8: Cost Optimization Audit

```bash
# 1. Use Infracost to see cost of current infrastructure
infracost breakdown --path .

# 2. Run plan with cost diff for proposed changes
infracost diff --path . --compare-to=infracost-base.json

# 3. Use AWS Cost Explorer + tagging strategy
# All Terraform-managed resources should have ManagedBy=terraform tag
```

### Scenario 9: Provider Version Upgrade

```bash
# 1. Read provider changelog for breaking changes
# 2. Update version in versions.tf
# 3. Run init -upgrade
terraform init -upgrade

# 4. Run plan in non-prod first
terraform plan

# 5. Address any deprecation warnings
# 6. Apply to dev → staging → prod with monitoring
```

### Scenario 10: A Resource Got Deleted Outside Terraform

```bash
# Plan will show the resource needs to be created
terraform plan
# +  aws_instance.web will be created

# If you want it back, just apply
terraform apply

# If you want Terraform to forget it (don't recreate):
terraform state rm aws_instance.web
```

### Scenario 11: Refactoring with Moved Blocks
Without `moved` blocks, refactoring causes destroy+recreate. Use `moved` to preserve resources:

```hcl
moved {
  from = aws_instance.web
  to   = module.compute.aws_instance.web
}

moved {
  from = aws_instance.workers[0]
  to   = aws_instance.workers["primary"]
}
```

### Scenario 12: Multi-Region Active-Active

```hcl
locals {
  regions = {
    primary   = "ap-south-1"
    secondary = "us-east-1"
  }
}

provider "aws" {
  region = local.regions.primary

  alias  = "secondary"
}

module "primary_stack" {
  source = "./modules/regional-stack"

    aws = aws.primary
  }
}

  source = "./modules/regional-stack"

    aws = aws.secondary
```

## 16. Troubleshooting Playbook

### Error: "Backend configuration changed"
```bash
terraform init -reconfigure
# or migrate state if intentional:
terraform init -migrate-state
```

### Error: "Provider produced inconsistent final plan"
Usually a provider bug. Try:
```bash
terraform init -upgrade
# If persists, add an ignore_changes lifecycle block as workaround
```

### Error: "Resource already exists" during apply
The resource exists in cloud but not in state. Import it:
```bash
terraform import aws_s3_bucket.logs my-existing-bucket
```

### Error: State Lock Stuck
```bash
# Check who holds the lock
terraform plan
# Force unlock if confident no one is running
terraform force-unlock <LOCK_ID>
```

### Error: "Error: Saved plan is stale"
Someone applied changes between your plan and apply. Re-plan:
```bash
terraform plan -out=tfplan
terraform apply tfplan
```

### Error: Cycle in dependency graph
Two resources depend on each other. Refactor — usually means you need a third resource or `depends_on` is misused.

### Slow plans
- Reduce blast radius — split state
- Increase parallelism: `terraform plan -parallelism=20`
- Target specific resources during dev: `terraform plan -target=module.app`

### Provider authentication failures
```bash
# AWS — check credential chain
aws sts get-caller-identity

# Set region explicitly
export AWS_REGION=ap-south-1
export AWS_PROFILE=production

# Enable debug logs
TF_LOG=DEBUG terraform plan 2>&1 | grep -i auth
```

### State Corruption Recovery
```bash
# State files have automatic backups
ls terraform.tfstate*

# Restore from backup
cp terraform.tfstate.backup terraform.tfstate

# For remote state, use bucket versioning to roll back
aws s3api list-object-versions --bucket company-terraform-state \
  --prefix prod/terraform.tfstate
```



## 17. Best Practices

### Code Organization
- **One module, one responsibility** — VPC module shouldn't create EC2 instances
- **Compose over configure** — small modules combined > one giant configurable module
- **Pin everything** — Terraform version, provider versions, module versions
- **Use `.terraform-version` file** for version managers
- **Always include README.md** in modules with usage examples

### Naming Conventions
```hcl
# Good
resource "aws_instance" "web_server" {}
resource "aws_db_instance" "primary_postgres" {}

# Bad
resource "aws_instance" "instance1" {}
resource "aws_instance" "ec2" {}
```

### Tagging Strategy
```hcl
locals {
  required_tags = {
    Environment = var.environment
    Project     = var.project
    Owner       = var.team
    ManagedBy   = "terraform"
    CostCenter  = var.cost_center
    Repo        = "github.com/company/infrastructure"
  }
}

provider "aws" {
  default_tags {
    tags = local.required_tags
  }
}
```

### Use Data Sources for Lookups
```hcl
# Instead of hardcoding AMI IDs
data "aws_ami" "amazon_linux_2" {
  most_recent = true
  owners      = ["amazon"]
  
  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

resource "aws_instance" "web" {
  ami = data.aws_ami.amazon_linux_2.id
}
```

### Lifecycle Rules
```hcl
resource "aws_instance" "critical_db" {
  # Prevent accidental deletion
  lifecycle {
    prevent_destroy = true
    
    ignore_changes = [
      ami,        # Managed by patching automation
      user_data,  # Bootstrapped externally
    ]
    
    create_before_destroy = true
  }
}
```

### Always Plan Before Apply (in production)
- No `apply -auto-approve` from a human's terminal in production
- CI/CD can auto-apply, but only after PR review
- Save plans to files for high-risk changes

### Backup Before Destructive Operations
```bash
# Snapshot state
terraform state pull > "backup-$(date +%Y%m%d-%H%M%S).tfstate"

# Database snapshots before schema changes
# AMI snapshots before instance changes
```



## 18. Common Pitfalls

### Pitfall 1: Using count when you should use for_each
```hcl
# ❌ Removing "bob" from the list shifts indices and recreates "charlie"
variable "users" {
  default = ["alice", "bob", "charlie"]
}

resource "aws_iam_user" "team" {
  count = length(var.users)
  name  = var.users[count.index]
}

# ✅ Each user has a stable identity
resource "aws_iam_user" "team" {
  for_each = toset(var.users)
  name     = each.key
}
```

### Pitfall 2: Storing state locally for team projects
Local state means whoever runs apply last wins, silently. Always use remote state with locking.

### Pitfall 3: Ignoring provider version constraints
```hcl
# ❌ Will break unpredictably as providers release
provider "aws" {
  region = "ap-south-1"
}

# ✅ Pin to compatible range
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

### Pitfall 4: Manually editing state files
Almost always a bad idea. Use `terraform state` subcommands.

### Pitfall 5: Putting secrets in tfvars files committed to git
Use `.gitignore` for `*.tfvars` and document required variables in `example.tfvars`.

### Pitfall 6: Not tagging resources
Untagged resources are invisible in cost reports. Make tagging mandatory via default_tags and validation.

### Pitfall 7: Massive monolithic state
A single state file with 5000 resources takes forever to plan. Split early.

### Pitfall 8: Hardcoding values
```hcl
# ❌
resource "aws_instance" "web" {
  ami = "ami-0c55b159cbfafe1f0"  # Region-specific, will break
}

# ✅
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]  # Canonical
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-22.04-amd64-server-*"]
  }
}
```

### Pitfall 9: Forgetting to lock provider versions
The `.terraform.lock.hcl` file should be committed to git. It pins exact provider versions and checksums.

### Pitfall 10: Not separating workspaces by blast radius
Production database changes shouldn't share a state file with experimental dev resources.

## 20. Migration and Adoption

### Adopting Terraform in an Existing Org

**Phase 1: Foundations (Month 1)**
- Pick a low-risk, greenfield project
- Set up remote backend with state locking
- Establish naming and tagging conventions
- Create CI pipeline for fmt/validate/plan

**Phase 2: Pattern Library (Month 2-3)**
- Build a few core modules (VPC, ECS/EKS cluster, RDS, IAM)
- Document module usage with examples
- Create a private module registry (Terraform Cloud or Git)

**Phase 3: Migration (Month 4-12)**
- Import existing critical infrastructure
- Don't try to migrate everything — focus on what changes often
- Establish boundaries: what's managed by Terraform vs what isn't
- Train teams; designate Terraform champions

**Phase 4: Maturity (Year 1+)**
- Policy as code (Sentinel/OPA)
- Cost gates in PRs (Infracost)
- Drift detection alarms
- Regular DR drills

### Migrating from Other IaC Tools

**From CloudFormation:**
- Use `former2` to scaffold Terraform from existing CFN
- Import resources one stack at a time
- Run both in parallel during transition (use lifecycle ignore_changes)

**From Pulumi/CDK:**
- Both can output Terraform-compatible state in some cases
- Generally easier to rewrite in Terraform than auto-translate

**From Ansible (for infra portions):**
- Ansible's `community.general` and cloud collections can output JSON state
- Manually rewrite as Terraform; keep Ansible for configuration management

### When NOT to Use Terraform
- **One-off ephemeral resources** — shell scripts may suffice
- **Pure Kubernetes app deployments** — Helm/ArgoCD/Kustomize are better
- **Configuration inside running servers** — use Ansible/Chef/cloud-init
- **Things with no API** — Terraform needs a provider
